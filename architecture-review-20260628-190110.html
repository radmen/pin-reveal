Deepening opportunities

The crypto core (src/derive.ts) is small and faithful to the frozen v1 contract. All the friction sits in src/App.tsx — a 1286-line component that owns the phase state machine, the worker handshake, the IndexedDB lifecycle, the theme system, and the markup for three screens. Five candidates below; the top recommendation is at the bottom.

Vocabulary: master key, label, login fingerprint, label fingerprint, derivation contract v1 per docs/pin-derivation-spec-v2.md. No CONTEXT.md or ADRs in repo.

Candidate 01 · state machine

Phase as a discriminated union, not a flat bag of fields

Strong
Files
src/App.tsx (lines 13–61, 113–243)
Recommendation strength
Strong — encodes invariants the spec calls out by name.

Problem

State is a 20-field flat record. Most fields are only valid in one phase (loginFp only in login, segments/revealIndex only in reveal). Every callsite must remember which subset applies. The spec's §9 invariant — “a fingerprint is on screen only while the form is clean” — is enforced by hand at six different set({...}) sites that all blank loginFp/labelFp/pin. Miss one site → silent bug, exactly the class the spec flags. The interface (the State shape) is shallow: it advertises every field equally regardless of when they're meaningful.

Solution

Replace the flat record with a discriminated union: LoginState | LabelState | RevealState. Each variant carries only its own fields. The "fingerprint blanked on edit" invariant becomes structural — editing username/password constructs a fresh LoginState with fp: null rather than mutating one of many fields. Use a small reducer (or two: one per phase) keyed on the variant tag — orchestration logic reads like state-chart edges.

Benefits

  • Locality: the §9 invariant lives in one place (the LoginState constructor) instead of being scattered across input handlers.
  • The interface is the test surface: tests can construct each phase's state directly and assert transitions, without rendering the whole component.
  • Deletion test passes: the union concentrates invariants that are currently smeared across App.tsx. Removing it would force re-spreading the same checks back across handlers.

Candidate 02 · seam

Extract a DerivationSession module

Strong
Files
src/App.tsx (113–234), src/derive.ts (98–128), src/derive.worker.ts
Recommendation strength
Strong — the App component shouldn't know about workers, IDB, race tokens.

Problem

Three concerns leak through App.tsx's body: (1) Worker construction + postMessage + terminate open-coded inside generateLogin (lines 147–154); (2) a token.current counter bumped at 5+ sites to discard stale derivations; (3) two refs (masterKey, pendingKey) that mirror an in-spec lifecycle (derived → pending → persisted via storeKey). The component must know about all three to coordinate one sentence from the spec: "derive a key, show its fingerprint, persist it on Proceed, forget on Logout."

Solution

Introduce a deep module with a four-method interface: 

session.start(username, password)  // runs Argon2id off-thread,
                                   // returns { fp } or null if superseded
session.confirm()                  // persists current pending key
session.derive(label, length)      // returns { fp, pin }
session.forget()                   // clears IDB + in-memory key

Behind the seam: worker lifecycle, snapshot-tagged race discard (per spec §9 wording), masterKey/pendingKey refs, IndexedDB calls.

Benefits

  • Depth: the interface is 4 methods; the implementation hides workers, IDB, refs, and race discard. Classic deepening.
  • Leverage: tests for the session can use a fake-time clock and assert "edit during derive → result discarded" without rendering Preact.
  • Spec alignment: swap the counter for a snapshot per "tag each run with its input snapshot and only display if the fields still match" — this lives naturally in the session.

Candidate 03 · code mass

Move inline styles to style.css

Worth exploring
Files
src/App.tsx (63–111, 256–372, ~700 lines of inline styles), src/style.css
Recommendation strength
Worth exploring — code-mass win, not a deepening.

Problem

Two ~25-property style dicts (DARK, LIGHT) + dozens of inline style objects + 4 style-helper functions (btnPrimary, lenBtn, rtBtn, segBox) bloat the component file. fontFamily: "'Space Mono',monospace" appears 23 times. CSS variables are already used — but they're created in JS and spread as inline style. The platform feature (CSS classes + variable scoping) is exactly what the project is already half-using.

Solution

Two classes on <body> (.theme-dark, .theme-light) define the variables. All inline styles become CSS classes in style.css. Style-helper functions (btnPrimary etc.) become BEM-ish class names like .btn-primary[disabled] / .seg.--active.--shown. App.tsx shrinks ~500 lines.

Benefits

  • Code-mass: ~500 lines drop. Less for an AI to scroll past to find the orchestration.
  • Native platform: theme via class on root + CSS vars is the browser-native pattern; the current setup reimplements it in JS.
  • Honest: this doesn't deepen anything. It's mass reduction. Worth doing once #1/#2 land, not before.

Candidate 04 · testability

Pin the frozen contract with vector tests

Strong
Files
src/derive.ts, src/App.test.tsx (the only test, 14 lines), src/test/setup.ts
Recommendation strength
Strong — the spec demands it: §0 says any silent change re-derives every PIN.

Problem

derive.ts is already a deep module — a frozen contract behind a 6-function surface. But there is one test in the entire repo, and it asserts a heading is rendered. None of the contract is pinned. The spec §12 lists 9 items "changing which re-derives outputs and forces manual rotation." A single accidental change to the normalization order, the digit threshold, the salt prefix, or the HMAC message format would silently corrupt every PIN — and no test would fail. Friction here is not "hard to test" — it's "nothing forces anyone to test."

Solution

Add a derive.test.ts with one vector per frozen item: known (password, username) → known master key bytes (computed once, committed); known label → known fingerprint words; known (label, length) → known PIN; the normalization examples from spec §5 table. Plus a @noble/hashes version-lock test (read package.json, assert range).

Benefits

  • Locality: the spec's §12 checklist becomes executable — one test per checklist item, named after the item.
  • The interface is the test surface: derive.ts's exports map 1:1 to test files, which is how a deep module should look.
  • Catches the most expensive bug class: silent contract drift. The spec literally warns about this in bold.

Candidate 05 · screens

Split App.tsx into three screen modules

Speculative
Files
src/App.tsx → App.tsx + LoginScreen.tsx + LabelScreen.tsx + RevealScreen.tsx
Recommendation strength
Speculative — only worth it after #1 lands; otherwise you split a shared bag of state across files and make it worse.

Problem

Three screens live in one 1286-line file. Each screen is ~250 lines of mostly-independent JSX. But — deletion test fails on its own: splitting today just spreads the flat State across three files via prop drilling, which is worse shallowness.

Solution

Wait for #1 (discriminated union). Then each screen receives only its own variant (props: LoginState) and a small dispatcher. Three small modules, each with a narrow interface, each independently render-testable.

Why it's speculative on its own

Two adapters of the same shape = a real seam; one = a hypothetical one. Right now there's one consumer (App.tsx). The win comes from #1 supplying typed contracts at the prop boundary. List it because someone will propose it; flag that it's a follow-on, not a starting point.

Top recommendation

Start with #1 (Phase as a discriminated union)

It's the keystone. It directly encodes the spec's §9 invariant (“fingerprint on screen only while form is clean”) into the type system instead of leaving it as a manual discipline at six callsites. It also unblocks #5 (per-screen modules need typed props) and naturally pairs with #2 (the session returns variants of the union as transitions). #4 stands alone and can land in parallel — the contract tests don't depend on the component shape at all.

A sensible order: #1 → #4 (parallel) → #2 → #5 → #3. #3 (CSS extraction) last — it's a code-mass win, not a deepening, and doing it earlier locks in styling decisions that #5 might want to relocate to per-screen modules.
